# PaperForge 研究会话产品架构设计

日期：2026-05-19

## 目标

PaperForge 不应该从空白建模表单开始。用户来到产品时，通常并不知道明确的研究方向、模型机制、变量、假设和可求解结构。产品真正要解决的是：把模糊研究兴趣推进成可写论文、可建模型、可求解析均衡的研究方案。

因此，PaperForge 的主体验应该是“博弈论建模导师 + 结构化研究工作台”。它先帮助用户发现方向，再帮助用户选择和调整模型方案，最后进入符号均衡求解和性质分析。

现有 Hotelling 工作台可以作为后续结构化成果面板继续复用，但不应该是用户输入粗糙想法后的第一界面。

## 当前问题诊断

现在的产品体验更像计算器：

- 用户输入一个粗糙想法。
- 产品很快进入工作台。
- 工作台出现大量空字段。
- 用户必须自己知道研究方向、效用函数、利润函数、假设和策略变量。
- 产品只有在用户已经完成大部分建模思考之后才开始有用。

理想体验应该更接近一次建模讨论：

- 用户说：“我想研究二手交易平台相关模型。”
- 系统提出多个研究方向，例如佣金与补贴策略、质量认证、平台治理强度、信息披露。
- 用户选择“佣金与补贴策略”。
- 系统提出最小可求解模型，并说明可加扩展项。
- 用户补充“两侧都可能收费或补贴”。
- 系统调整为双边净收费模型，生成效用函数、需求推导、利润函数、均衡求解路径和可复用符号代码。

## 核心原则

1. 方向发现先于模型表单。
2. 产品应该提出、比较、推荐，而不是只收集字段。
3. 用户主要做选择、否定、补充和调整。
4. 项目初始阶段不应该出现空白工作台。
5. 对话产生的研究内容要沉淀成结构化资产。
6. 探索记录和正式项目必须区分。
7. 第一版用户自带 API Key 只保存在本地浏览器，不上传服务器。
8. UI 应该像研究产品，不像泛聊天工具，也不像参数表格。

## 页面架构

### 1. 首页 `/`

首页负责品牌、定位和入口，不放研究想法输入框。

第一屏建议：

- 顶部导航：
  - PaperForge
  - 功能
  - 文档
  - 登录
  - 注册
- 主标题：
  - 从模糊选题到可求解的博弈论模型
- 副标题：
  - 面向 Hotelling、双边平台和符号均衡推导的研究工作台。
- 主按钮：
  - 开始研究
- 次按钮或链接：
  - 查看示例
  - 了解工作流

首页下方可以展示：

- 方向发现
- 模型共创
- 符号均衡
- 性质分析
- 示例流程：
  - 二手交易平台 -> 佣金与补贴策略 -> 双边净收费 Hotelling 模型 -> 均衡求解 -> 性质分析

定价页第一版不做。导航里不放“定价”，避免出现空页面或虚假商业化信息。

### 2. 登录与注册

用户点击“开始研究”后，如果未登录，进入登录或注册。

原因：

- 研究会话需要保存。
- 探索记录需要可回看。
- 模型来源设置依赖用户身份。

### 3. 启动向导 `/launch`

登录后进入一个轻量两步向导。

#### 第一步：选择模型来源

选项一：使用 PaperForge 提供的模型

- 用户无需配置。
- 第一版不做真实收费。
- 文案可以写“测试阶段可用”或“由 PaperForge 托管”。

选项二：使用自己的模型

- API Key 第一版只保存在当前浏览器。
- API Key 不保存到 PaperForge 服务器。

支持的服务商模式：

- OpenAI
- Anthropic
- OpenAI-compatible
- Anthropic-compatible

自带模型需要的字段：

- 服务商模式
- API Key
- Base URL
  - 官方 OpenAI 和 Anthropic 可选。
  - compatible 模式必填。
- 模型名
- 测试连接按钮

安全提示：

> API Key 仅保存在当前浏览器，不会上传到 PaperForge 服务器。更换设备或清除浏览器数据后需要重新填写。

#### 第二步：输入研究想法

研究想法输入框放在这里，而不是首页。

示例占位：

> 例如：我想研究二手交易平台的收费和补贴策略

主按钮：

- 开始生成研究方向

提交后，系统创建一条研究探索记录。

### 4. 研究空间 `/research`

研究空间是核心产品页面，采用三块区域。

#### 左侧：研究记录

左侧是记录导航。

记录类型：

- 探索记录
- 正式项目

行为：

- 每次用户开始生成研究方向，新增一条探索记录。
- 探索记录默认保存到账户数据库。
- 探索记录可以重新打开、继续讨论、重命名、删除。
- 探索记录不会自动变成正式项目。
- 用户点击“采用这个方向”后，探索记录升级为正式项目。

这样符合真实使用方式：用户可能会探索很多想法，但只有一部分会进入正式论文项目。

#### 中间：建模导师对话

中间是主要思考区域。

它负责：

- 方向发现
- 追问澄清
- 用户补充和修正
- 模型方案比较
- 效用函数讨论
- 需求推导解释
- 均衡求解解释
- 符号失败解释

这个区域应该保留网页端大模型对话的自然感，但不能只是泛聊天。消息应该有任务类型，例如：

- 方向建议
- 澄清问题
- 模型推荐
- 模型修订
- 求解步骤
- 符号警告
- 可采用成果

#### 右侧：研究资产面板

右侧不是完整内容容器，而是当前阶段的摘要、决策面板和资产索引。

它只显示当前最重要的 3-5 个对象，详细内容在中间展开。

右侧负责：

- 显示当前阶段。
- 显示当前推荐成果。
- 显示关键结构化字段。
- 显示警告和可求解性状态。
- 提供关键行动按钮。
- 支持锁定重要字段，防止后续对话擅自改乱。

右侧不能变成拥挤的杂物栏。

## 研究工作流

### 阶段一：方向发现

输入：

- 模糊研究想法。

输出：

- 3-5 个候选研究方向。

每个候选方向包含：

- 研究故事
- 现实背景
- 模型机制
- 适合的模型类型
- 主要参与者
- 策略变量
- 可能的均衡目标
- 可做的性质分析
- 可求解性评级：高 / 中 / 低
- 风险：
  - 符号爆炸
  - 自由参数太多
  - 创新不足
  - 假设难以解释
- 推荐程度

示例：

用户输入：

> 我想研究二手交易平台相关的模型

系统可以给出：

- 佣金与补贴策略
- 质量认证机制
- 平台治理强度
- 信息披露与卖家类型

系统应该推荐最适合作为起步模型的方向，并解释原因。

### 阶段二：采用方向

用户可以点击“采用这个方向”。

结果：

- 探索记录升级为正式项目。
- 被采用的方向成为项目研究前提。
- 对话进入模型共创阶段。

### 阶段三：模型共创

采用方向后，不进入空白工作台。

系统先提供：

- 推荐的最小可求解模型。
- 可选扩展项。
- 可求解性说明。

推荐交互：

- 默认给出最小模型。
- 同时列出以后可以加入的扩展项。
- 用户可以通过对话或勾选结构化控件调整。

以二手交易平台为例：

推荐最小模型：

- 两个竞争平台。
- 买家侧和卖家侧。
- 两侧都可以收费或补贴。
- 价格变量是净收费，可以为正或负。
- 线性跨边网络效应。
- 买家和卖家两侧都采用 Hotelling 差异化。

可选扩展：

- 买家服务成本。
- 卖家审核成本。
- 质量认证。
- 交易保障强度。
- 卖家类型或信息披露。
- 平台非对称。

### 阶段四：生成模型资产

模型版本被接受后，系统生成结构化资产：

- 研究问题
- 参与者
- 策略变量
- 时序
- 符号字典
- 效用函数
- 需求推导
- 利润函数
- 假设条件
- 可求解性检查
- 模型设定草稿

用户可以检查和编辑这些资产，但它们应该来自前面的讨论和决策，不应该从空白开始。

### 阶段五：均衡求解

系统基于已确认的模型资产进行符号均衡求解。

输出：

- 求解目标
- 需求方程
- 一阶条件
- 需要时给出反应函数
- 可得到时给出闭式均衡
- 存在性或内部解条件
- 推导说明
- 可复用 SymPy 代码

硬规则：

- 数值均衡不能替代符号均衡。

失败处理：

- 如果符号求解失败或爆炸，系统说明原因。
- 系统给出简化方向，例如对称平台、减少参数、线性化网络效应、分阶段求解。

### 阶段六：性质分析

有均衡结果后，系统建议可做的性质分析。

输出：

- 候选命题
- 符号导数或符号差值
- 符号条件
- 阈值条件
- 证明思路
- 经济直觉

硬规则：

- 数值比较静态不能替代符号性质分析。

## 右侧研究资产面板状态

### 方向阶段

右侧显示：

- 候选方向列表
- 推荐标签
- 可求解性状态
- 采用这个方向按钮

详细解释在中间展开。

### 模型阶段

右侧显示：

- 当前模型版本
- 主要参与者
- 策略变量
- 关键假设
- 可求解性状态
- 已锁定字段
- 确认模型并生成效用函数
- 进入均衡求解

效用函数和利润函数详细内容在中间展开。

### 均衡阶段

右侧显示：

- 求解状态
- 是否得到闭式解
- 关键均衡表达式摘要
- 条件数量
- 警告
- 进入性质分析

完整推导和代码在中间展开。

### 性质分析阶段

右侧显示：

- 候选命题
- 已选择参数
- 符号条件或阈值状态
- 警告
- 生成证明思路
- 导出分析草稿

完整符号推导在中间展开。

## 数据保存策略

### 保存到账户数据库

- 探索记录
- 正式项目
- 对话历史
- 结构化研究资产
- 已采用方向
- 模型选择
- 均衡结果
- 性质分析结果

### 第一版仅保存在本地浏览器

- 用户自带 API Key

### 可保存到数据库的非敏感配置

- 服务商模式
- 模型名
- 是否使用 PaperForge 提供模型

API Key 不保存到服务器。

## 模型来源配置

第一版支持：

- PaperForge 提供模型。
- 用户自带模型。

用户自带模型支持：

- OpenAI
- Anthropic
- OpenAI-compatible
- Anthropic-compatible

compatible 模式用于覆盖大多数第三方模型服务。

模型来源后续应该能在设置页或账户菜单中修改。

## 需要的页面

### `/`

首页。

### `/sign-in` 和 `/sign-up`

登录与注册。

### `/launch`

两步启动向导：

1. 选择模型来源。
2. 输入研究想法。

### `/research`

研究空间，三块区域：

- 研究记录。
- 建模导师对话。
- 研究资产面板。

后续可以扩展为 `/research/[sessionId]`。

### `/projects/[id]`

正式项目页面。

可以复用研究空间布局，但左侧记录和右侧资产更偏正式项目。

### `/settings/model`

模型来源设置。

第一实现切片可以先只做启动向导中的配置，设置页后置。

### `/docs`

简单文档页，解释：

- 支持什么工作流。
- 为什么均衡必须是符号解。
- 模型来源如何配置。
- 如何使用 Hotelling 和双边平台建模流程。

## UI 方向

### 首页

- 像一个聚焦的研究产品官网。
- 第一屏冷静、直接、可信。
- 不使用夸张装饰、泛 AI SaaS 渐变、无意义大图。
- 主按钮清楚引导“开始研究”。

### 研究空间

- 更像 agentic research IDE，而不是表单向导。
- 左侧记录列表像研究线程。
- 中间对话区域要可读、可长时间使用。
- 右侧资产面板要紧凑、决策导向。
- 数学内容和代码块要清晰、可复制。
- 空状态必须告诉用户下一步该做什么。

## 本次重构切片不包含

- 真实收费或付费计划。
- 公开分享。
- 多人协作。
- 完整论文编辑器。
- 数值仿真。
- 服务器保存用户 API Key。
- PDF 文献完整解析。
- 完整导出链路。

## 已确定决策

1. 首页不放研究想法输入框，只放“开始研究”主按钮。
2. 点击开始后，未登录用户先登录或注册。
3. 登录后进入两步启动向导：模型来源，研究想法。
4. 模型来源支持 PaperForge 托管和用户自带模型。
5. 用户自带模型支持 OpenAI、Anthropic、OpenAI-compatible、Anthropic-compatible。
6. API Key 第一版只保存在本地浏览器。
7. 生成研究方向后进入三栏研究空间。
8. 左侧记录先是探索记录，采用方向后升级为正式项目。
9. 探索记录和正式项目都保存到账户数据库。
10. 中间是建模导师对话。
11. 右侧是研究资产摘要和决策面板，不承载全部详细内容。
12. 采用方向后，系统先给推荐的最小可求解模型和可选扩展项。

## 待实施计划中继续拆分的问题

1. 模型来源设置页是否第一切片就做，还是先只做启动向导。
2. 探索记录和正式项目使用同一张数据库表还是分表。
3. 对话如何同步生成结构化资产，是每轮消息后更新，还是用户点击“采用/确认”后更新。
4. 第一实现切片是否同时支持 OpenAI-compatible 和 Anthropic-compatible 调用，还是先做一个 compatible 路径。
5. 旧 Hotelling 工作台中哪些组件复用，哪些废弃。

4. 三栏研究空间布局。
